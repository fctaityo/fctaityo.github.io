# Review Agent文体誤検知とfixed未反映

- Review ID: `REV-20260718-003`
- 日付: 2026-07-18
- Status: `Open`
- Review Type: 打鍵レビュー
- 対象: Review Stage

## 結果

文体として「です・ます調禁止」を指定して記事生成を実施した。

Writing本文は常体で書かれていたが、Review Agentは文章の硬さや学術的な印象を「です・ます体」と誤認し、Tone不一致を指摘した。

さらにReview Agentは`fixed`と判定した一方で、修正済み本文を返していなかった。後続工程はWriting本文をそのまま採用したため、Reviewの宣言と実際の成果物に不整合が生じた。

## 期待結果

- 文体判定を文章の印象ではなく、実際の文末形式に基づいて行う。
- `fixed`を返す場合は、修正済み成果物を返す。
- 修正していない場合は、未修正を示す判定を返す。
- 後続工程が、宣言だけでなく修正成果物の存在を確認する。

## 実際結果

```text
User tone:
です・ます調禁止

Writing:
常体

Review:
です・ます体と判定
verdict = fixed

Review artifact:
修正済み本文なし

Final Artifact:
Writing本文を採用
```

## Root Cause

### 1. 文体と文章印象の混同

Review Agentは「硬い」「丁寧」「学術的」といった文章全体の印象を、文法上の「です・ます体」と混同した。

常体で使われる「である」「となる」「求められる」なども、丁寧な文章として誤認した可能性が高い。

### 2. `fixed` Contractの不足

`fixed`という判定だけが許容され、次の条件がContract Gateで検証されていなかった。

- 修正済み成果物が存在する。
- 元成果物と修正版が異なる。
- 修正内容の説明が存在する。
- 後続工程が修正版を採用する。

その結果、「修正したと宣言しただけ」の状態で処理を継続できた。

## 影響範囲

- 文体指定を伴う記事生成
- Review Agentが`fixed`を返すケース
- Review判定と成果物の採用元が分離しているWorkflow
- Reviewの宣言だけを信頼するFinal Audit

今回の誤検知では本文への直接被害は限定的だった。

しかし、事実誤認やTopic逸脱について同じ不整合が発生した場合、未修正成果物が正常扱いされる可能性がある。

## 改善方針

### 1. 文体判定ルールを明示する

文体は文章の印象ではなく、文末形式で判定する。

```text
です・ます体:
です、ます、でした、ません、でしょう など

常体:
だ、である、する、した、となる、求められる など
```

「硬い」「学術的」「回りくどい」は、文体とは別の品質項目として扱う。

### 2. `fixed` Contractを強化する

`fixed`の場合は、最低限次を必須とする。

- 修正済み成果物が存在する。
- 元成果物と修正版が異なる。
- 適用した修正内容が記録されている。
- 指摘内容と成果物差分が対応している。

条件を満たさない場合は、Review Contract違反として停止する。

### 3. verdictの責務を分離する

判定を次のように整理する。

- `passed`: 問題なし
- `needs_fix`: 問題あり、未修正
- `fixed`: 問題を検出し、修正版成果物を返却済み
- `blocked`: 修正不能または重大な根拠不足

### 4. Final Auditで整合性を確認する

Final Auditは、Reviewの宣言だけでなく次を確認する。

- Review verdict
- 修正版成果物の有無
- 採用された本文の出所
- 修正前後の差分
- 未解決issueの残存

## Lessons Learned

Review Agentの判定は、成果物の存在と一致して初めて意味を持つ。

「fixed」という文字列は修正の証拠ではない。修正版成果物、差分、採用元をContractとして検証する必要がある。

## 判定

`Open`

文体判定ルールと`fixed` Contractを修正し、同一条件で回帰試験を実施する。
