## Workflow Success偽装事件

### 症状

ワークフローはエラーなく最後まで完走し、実行ステータスも成功になった。

しかし、生成された記事は途中で切れていたり、結論セクションが欠落していたり、レビュー結果が最終成果物へ反映されていなかった。

実LLMを使った記事生成では、本文が2394文字付近で終了し、予定していた内容の途中で切れていた。

それでもReviewは通過し、Package Outputまで保存され、ワークフロー全体はSuccessとして終了した。

ワークフローエンジン上では成功しているため、成果物の破損が正常終了の裏側に隠れた。

### Root Cause

Workflow SuccessとArtifact Successを同一視していた。

ワークフローが例外なく完走したことを、そのまま成果物が正しく成立した証明として扱っていた。

記事の文字数、必須セクション、結論、文末の成立などを、成果物側の成功条件として十分に検証していなかった。

### 教訓

Workflow SuccessとArtifact Successは別物である。

Workflow Successは、ワークフローエンジンが例外なく完走した状態にすぎない。

Artifact Successは、契約と成果物検証を通過し、期待される成果物が有効に成立した状態である。

前者は後者を保証しない。

全部のノードが成功していても、成果物が完成しているとは限らない。

## Retry無限保険化事件

### 症状

契約違反や生成失敗が起きた際、成功するまで再生成を繰り返せばよいという発想に寄りかけた。

Retryが、失敗から有限回だけ復旧を試みる手段ではなく、最終的な成功を保証するための保険として扱われそうになった。

Research出力のJSON構文違反に対してRetryを実行したが、再生成後も初回と同型の構文違反が発生した。

Retry経路が存在していても、失敗原因や再生成条件が変わっていなければ、同じ壊れ方を再生産する。

### Root Cause

Retryの上限、停止条件、失敗時の扱いが設計上の責務として固定されていなかった。

再生成すればいつか通るという期待が、壊れたデータを正しく止める判断より優先されていた。

初回失敗のRaw出力や不要な文脈をRetry入力へ引き継ぎ、同じ失敗条件を再現する危険も十分に制御されていなかった。

### 教訓

Retryは保険ではない。

Retryは上限を明示した有限の復旧手段である。

Retry入力は、失敗した出力を丸ごと再投入するのではなく、再生成に必要な情報と違反内容へ限定する。

再生成後も同じContract Gateを通す。

再生成後も契約を満たさない場合は、壊れたデータを後段へ流さず、その場所で正しく停止する。

## Review Agent本文上書き事件

### 症状

レビュー担当が、問題点の判定だけでなく本文の修正や再生成まで担いそうになった。

レビュー結果と修正後本文の両方を返させることで、どの本文が正本なのか分からなくなる危険が生じた。

実際の構成では、Review Agentが指摘対象だけでなく、問題のない箇所まで善意で書き換える可能性があった。

Writing Agentが生成した本文と、Review Agentが返した修正版本文が併存し、最終成果物がどちらを採用したのか追跡しにくい状態になった。

### Root Cause

Reviewの責務と、成果物を書き換える責務の境界が曖昧だった。

レビュー担当に善意の修正を許すことで、正本の所有者が複数になりかけた。

判定、修正指示、本文再生成を一つの工程へ集めていた。

### 教訓

Reviewは判定する。

本文を書き直させない。

修正が必要な場合は、違反内容を明示し、本文の所有者であるWriting工程へ戻す。

誰が正本を持つかは常に一つに保つ。

## AI劇団四季事件

### 症状

企画AI、調査AI、執筆AI、レビューAIが人格を持ち、「よろしくお願いします」「確認しました」と会話しながら作業を進める構成になりかけた。

見た目は分かりやすいが、AI同士が人間の組織コミュニケーションを演じること自体が設計の中心になっていた。

### Root Cause

人間同士の認識合わせを、そのままAI同士の連携方法として持ち込んでいた。

役割名や会話表現に注目し、入力形式、出力形式、責任境界、停止条件の固定が後回しになっていた。

### 教訓

AIチームは表現であって、設計ではない。

AI同士に必要なのは挨拶や相談ではなく、DTO、Contract Gate、責任境界、停止条件である。

役職ではなく工程で責任を切る。

## fixture成功＝実LLM成功事件

### 症状

fixtureを使ったユニットテストや統合テストが成功したことで、実LLMを使った連続正常系も成立したように見えた。

しかし、実際のLLM出力では契約違反、欠落、文字数超過、JSON破損などが発生し、同じ条件では安定しなかった。

### Root Cause

決定的なfixture入力による成功と、非決定的な実LLM出力による成功を同一視していた。

テスト対象の違いを、成功件数だけで丸めて評価していた。

### 教訓

fixtureでの成功と実LLMでの成功は同一ではない。

テストの成功は、何を入力し、何を検証した成功なのかまで含めて記録する。

## 完全始動＝完成事件

### 症状

改善サイクルが回り始めた段階を、「完成したシステム」と表現しそうになった。

未完成のGate、未安定な実LLM正常系、残された設計課題があるにもかかわらず、完成という言葉で進捗を丸める危険があった。

### Root Cause

始動と完成を区別していなかった。

動かし、壊し、記録し、直し、再び試せる状態になったことと、すべての契約と正常系が完成したことを混同していた。

### 教訓

完全始動は完成ではない。

完成したから始めるのではない。

壊し、直し、育て続ける改善サイクルが回り始めた時点を、始動と呼ぶ。

## Stage境界消失事件

### 症状

工程間で必要なDTOだけを渡さず、LLMが返したRaw出力を後段へそのまま渡していた。

Raw出力には、本来の成果物だけでなく、別Stageの識別情報や余分な文章が含まれていた。

その結果、後段のAgentが別工程のJSONを読み、自身が処理すべきStageを正しく識別できない事象が発生した。

Prompt上では役割を指定していても、入力に別Stageの構造が混入したことで責任境界が崩れた。

### Root Cause

DTOではなくRaw出力を工程間の受け渡し単位としていた。

各工程が、どの契約だけを入力として読むのか固定されていなかった。

Stage境界をPromptの説明へ依存し、データ構造によって強制していなかった。

### 教訓

Agentを信用する前に入力を固定する。

工程間でRaw出力を共有しない。

必要な情報だけをNormalizeしたDTOとして渡す。

Stageは役割名やPromptではなく、DTO境界とContract Gateで守る。

## 必須DTO欠落伝播事件

### 症状

Planning DTOの必須項目が欠落した状態でも、ワークフローが後段へ進んだ。

Researchは十分な調査条件を受け取れず、その影響がWritingやReviewなど後続工程へ連鎖した。

後段では、調査不足、論点不足、構成不良、レビュー違反など複数の異常として現れた。

表面上は複数の障害に見えたが、起点は上流DTOの単一欠落だった。

### Root Cause

必須項目の存在と型を確認するContract Gateが不足していた。

上流工程の不完全なDTOを、後段が補完できる前提で流していた。

欠落した情報を後段のAgentが推測や創作で埋める可能性も、責務違反として停止できていなかった。

### 教訓

一つの欠落は、一つの失敗として現れるとは限らない。

上流の単一欠落が、下流では複数の品質異常へ姿を変える。

入口で止めれば出口は壊れない。

必須項目が欠けたDTOは、その工程で停止する。

後段のAgentへ欠落補完を期待しない。

## 巨大JSON・E2BIG事件

### 症状

長文記事、調査結果、監査情報などを含む大きなJSONを、コマンドライン引数として後段処理へ渡した。

その結果、OS側の引数サイズ制限に達し、`spawn E2BIG` が発生した。

LLMによる生成処理は完了していたが、その成果物を次工程へ運ぶ段階で停止した。

モデルの能力やPromptとは無関係な場所で、パイプライン全体が失敗した。

### Root Cause

生成物の内容だけを設計し、データ量に応じたTransport方法を設計していなかった。

小さなDTOと長文成果物を、同じ方法で運搬できると考えていた。

コマンドライン引数を、大量データの受け渡し経路として使用していた。

### 教訓

生成とTransportは別の責務である。

LLMが生成できても、運べなければ成果物にはならない。

大きな成果物はファイルへ保存し、後段にはファイルパスや識別子を渡す。

データ形式だけでなく、データ量と運搬方法も設計対象に含める。

## Context Window 4096天井事件

### 症状

Promptでは十分な長さの記事を要求していたにもかかわらず、生成本文が2394文字付近で途中終了した。

文章表現やPromptの文字数指示を修正しても、同様の位置で生成が終了した。

確認したところ、Ollama Provider側とは別に、DifyのLLM Node側でContext Windowが4096に設定されていた。

Node側の設定が実行時の上限として作用し、Promptで長文を要求しても生成を継続できなかった。

### Root Cause

生成量の問題をPromptまたはモデルの能力だけの問題として調査していた。

モデル、Provider、LLM Nodeなど、複数の設定階層のうち、実行時にどの値が有効になるのか確認していなかった。

Runtime設定を設計・レビュー対象として扱っていなかった。

### 教訓

Promptを疑う前にRuntimeを見る。

Promptは上限を突破できない。

モデル設定、Provider設定、Node設定の優先関係を確認する。

設定値は存在するだけでは意味がない。

実行時に有効な値をEvidenceとして確認する。

Runtime設定も設計の一部である。
